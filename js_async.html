Javascript Async

I have tried to write this article without any typing errors. But if any typing error still exists, then sorry. I will fix that if I notice that anytime.

Let's start with a simple block of code :

function hello(){
	console.log("A block of code which is very important...");
}
console.log("Print at start");
hello();
console.log("Print at end");

Output of the above code :

Everything looks good till now because the code we are running is synchronous code. But suppose we want to run the hello() function after 5 secs. So we have to use the setTimeout() function which is an asynchronous function. So just have a look how it runs :

function hello(){
	console.log("A block of code which is very important...");
}
console.log("Print at start");
setTimeout(hello,5000);
console.log("Print at end");

Oh..What happened? The hello() function runs after 5 secs but the console.log("Print at end")executed before the end of hello() function. This is because whenever a async code is hit it runs in the background and all the other codes after that keep on executing as normal. But this was not what we wanted. We wanted that that hello should run after 5 secs and then console.log("Print at end") should have executed. So how to do this?

From here comes the concept of blocking code and non-blocking code. Hope these two words make sense.

So now we understood that we should not block the whole code after the delayed function but we should block the part of the code which we need to run after the delayed function has been completed. For this the concept of callbacks are very useful.

Actually callbacks means passing a function as an parameter of another function. So the function we are using in the setTimeout() function is also a callback function. And the function which accepts callback function in parameter are called higher-order functions.

So until now we got an idea that javascript code runs synchronously but as soon as it hits some asynchronous code like setTimeout etc. it sends it to the background and runs rest of the code.

Now we can conclude that again the code within the setTimeout function runs synchronously...!!

So if we create a function (create separately or create it anonymously its same) inside the setTimeout() function and move our console statement inside the function then it should run synchronously...Let's see the same example :

function hello(){
	console.log("A block of code which is very important...");
}

console.log("Print at start");
setTimeout(()=>{
    hello();
    console.log("Print at end");
},5000);

Output :

Yay... it runs well !! Actually this example doesn't require callback so much so let's see another example : 

function getStudent(callback){
    console.log("Fetching data");
    setTimeout(()=>{
        let name = "Souptik"; // Data fetched from database
        callback(name); 
    },5000); /* Simulating that the name of the student is fetched from 
    a database and due to network traffic takes 5 ses */
}

getStudent((student)=>{
    console.log(student);
});

So here the name of the student can only be printed after the data has been fetched otherwise the name would show undefined. Now you may ask that why send the callback function and receive the name as a parameter of the callback function and then print it. We can directly write the console statement inside the function of setTimeout. But this is only a console statement and in real world there would be huge chunks of code and writing them inside that same function would be very bad and also we will be destroying the concept of functional programming i.e pack different actions in function and use them as needed.

So by this hope concept of callbacks are clear and you would be able to write the previous code using callbacks.

Now everything looks good till now. But let make things a bit more complex. Here you have to do one task after the first have finished but suppose there are like this many tasks one after another let's see a few console log tasks in sequence :

setTimeout(()=>{
    console.log("First");
   setTimeout(()=>{
       console.log("Second");
       setTimeout(()=>{
           console.log("Third");
           setTimeout(()=>{
               console.log("Fourth");
           },50000);
       },5000);
   },5000);
},5000);
// Two three four these functions are some functions which are already defined
function one(two){
    setTimeout(()=>{
        two(()=>{
            three(()=>{
                four();
            });
        });

So this is the ugliness of callbacks. These are extremely simple things written but when the callback functions will be more complex big and increase in number the clumsiness and the right indentation will increase ultimately the readability of the code will be lost. This situation is called the callback hell.

Don't want hell? Then let's promise that we will understand promises....

let first = new Promise((resolve, reject)=>{
    console.log("Started working");
    setTimeout(() => {
        console.log("Work finished");
        let error = false; // true or false
        if (error == false){
            let name = "Souptik";
            resolve(name);
        }
        else{
            let errCode = "404";
            reject(errCode);
        }
    }, 5000);
});

first.then((name)=>{
    console.log("This is the resolve function passed inside then function.");
    console.log("Student name is " + name);
}).catch((errCode)=>{
    console.log("This is the reject function passed inside catch function.");
    console.log("Some error happened and the error code is : " + errCode);
});

Output if error = true :

This is the format of promises. The resolve and reject acts as the callback functions. If the resolve function is called then the function inside the then will be executed and if reject is called then the function inside the catch will be called.

Now lets see how the chaining can be done in a proper way and we can get out of the callback hell.

Lets define some functions which are to be sequenced but this tie returning promises : 

function getStudent(){
    return new Promise((resolve, reject)=>{
        setTimeout(()=>{
            console.log("Fetching student name....");
            console.log("Fetched student name");
            let name = "Souptik";
            resolve(name);
        },5000);
    });
}
function getClassStrength(std){
    return new Promise((resolve, reject)=>{
        setTimeout(()=>{
            console.log("Class :" + std);
            console.log("Fetching class strength.... of class " + std);
            console.log("Fetched class strength");
            let classStrength = "50";
            resolve(classStrength);
        },5000);
    });
}
function getClass(name){
    return new Promise((resolve, reject)=>{
        setTimeout(()=>{
            console.log("Student name : " + name);
            console.log("Fetching class..... of " + name);
            console.log("Fetched class");
            let std = "1st Year";
            resolve(std);
        },5000);
    });
}

The code will run in the following sequence : getStudent ‚Üí getClass ‚Üí getClassStrength. As we have used promises our code will be just this small :

getStudent()
    .then(name => getClass(name))
    .then(std => getClassStrength(std))
    .then(strength => console.log("His class strength : " + strength));

Just using then after then we can sequence the functions. Each function returns a promise so when each function is called, using then we can easily declare that what function needs to be executed next. At first look it might seem a bit complicated but a bit of practice will make it very easy.

Output of the above code :

Also if we have two functions returning Promises and want to run them together and have a common resolve functions for both and also run the resolve function only when both are finished we can use something like :

Promise.all([first1, first2]).then(()=>{//My resolve function});

So, all well and also we are not stuck at the call back hell.

Still can we make it more easier and sequence the functions without then and in synchronous style. So, the function declarations will be same. Only we can sequence our code like :

async function mySequencedCode(){
    let name = await getStudent();
    let std = await getClass(name); 
    let classStrength = await getClassStrength(std);
    console.log("His class strength : " + classStrength);

}
mySequencedCode();

The out put of the above will be the exactly same, only this is written in a much easier format. This is async-await (only available at Es8) concept.

So, that's it for Javascript async with a bit of practice this will be very helpful in many situations.
